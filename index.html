Phase 2

What Is SPL?

SPL — Search Processing Language — is how analysts ask questions of their security data. Think of it as Google for your logs.

Definition

SPL

Search Processing Language

The query language used to search, filter, and analyze data in Splunk.

🔍

Google

Python Course

Searches websites. Returns pages about Python courses.

📊

Splunk

EventCode=4624

Searches logs. Returns successful login events.

Without SPL

Analyst must manually scroll through millions of events. Impossible at enterprise scale. Threats go undetected.

With SPL

Analyst types a 3-second query and finds exactly what matters. Investigation time drops from hours to minutes.

Phase 4

Your First Searches

Let us run actual searches. Copy these queries into the Splunk Search Bar and observe the results.

Search Everything
index=*
Meaning: Show all indexed data from every index. Use this to see what data is available in the environment.
Search Windows Logs Only
index=windows
Meaning: Show events only from the Windows index — includes Security, System, and Application logs.
Search Linux Logs Only
index=linux
Meaning: Show events only from the Linux index — includes auth.log, syslog, and messages.
Search Security Events Only
source="WinEventLog:Security"
Meaning: Show only Windows Security Log events. This is the primary source for authentication events.

Phase 5

Search Filtering

Searching everything creates noise. Filtering removes noise. Precise filters are what separate beginner analysts from experienced investigators.

Successful Logins
EventCode=4624
Returns all successful login events. This is the most common search in SOC investigations.
Failed Logins
EventCode=4625
Returns all failed login events. Essential for identifying brute-force attacks.
Account Lockouts
EventCode=4740
Returns account lockout events. Often related to brute-force attacks or misconfigured services.
Specific User
user=Administrator
Returns all events involving the Administrator account. Use this to track a user's activity.
Specific Host
host=DC01
Returns all events from a specific system. Use this to investigate a particular server.
Combined Filter
index=windows EventCode=4624 user=Administrator host=DC01
Multiple filters combined. Returns only successful Administrator logins on DC01. This is how precise investigations work.
💻 Exercise

Progressive Filtering

1. Find all successful logins: EventCode=4624
2. Then filter to Administrator logins only: EventCode=4624 user=Administrator
3. Then narrow to Administrator logins on DC01: EventCode=4624 user=Administrator host=DC01

Notice: Each filter removes irrelevant data. You are narrowing in on exactly what matters.

Phase 6

Understanding Time in Investigations

Every SOC investigation starts with a question about time. When did something happen? Time is the most critical filter.

⚖ Scenario

A user reports suspicious activity: "I saw a login from my account at 10:15 AM, but I was in a meeting."

Should we search all logs — or only the relevant timeframe?

Answer: Search only the relevant timeframe. Set the Time Picker to a window around 10:15 AM (e.g., 10:00 AM – 10:30 AM). This eliminates 99% of irrelevant data and focuses the investigation.

Good analysts narrow the scope. Bad analysts search everything. Time is the most powerful filter you have.

Phase 9

Understanding False Positives

This is one of the most important concepts for new SOC analysts. Not every alert is an attack. Learning to distinguish noise from real threats is what makes an analyst valuable.

Scenario 1: A single failed login (4625) for user "john" at 10:00 AM from the office IP range. Normal or suspicious?
Correct — Normal. A single failed login is common. The user may have mistyped their password. One event alone is not an indicator of compromise.
Scenario 2: User "sarah" had 3 failed logins followed by a successful password reset. Sarah says she forgot her password. Normal or suspicious?
Correct — Normal. Password reset after forgotten password is a common and expected pattern. No indicators of an attack.
Scenario 3: 50 failed logins (4625) for "Administrator" from IP 203.0.113.50 in 2 minutes. Normal or suspicious?
Correct — Suspicious. 50 failures in 2 minutes targeting a privileged account from an external IP is a classic brute-force attack. This requires immediate investigation.
Scenario 4: User logged in at 3 AM from an IP in a different country. The user is on vacation. Normal or suspicious?
Correct — Suspicious. A login from a foreign country while the user is on vacation is highly anomalous. This could indicate credential theft. Requires user verification.

Alert ≠ Incident

Alert → Investigation → Conclusion

Every alert must be investigated before it can be classified as an incident or a false positive.

Phase 10

Basic SPL Operators

Now that you understand filtering, let us formalize the operators that make SPL powerful. These are the building blocks of every SOC investigation.

AND Both conditions must be true
index=windows EventCode=4624
Returns events where BOTH conditions are true: the event is from the Windows index AND it is a successful login.
✓ Windows + 4624 — returned
✗ Linux + 4624 — not returned
OR Either condition can be true
EventCode=4624 OR EventCode=4625
Returns events where EITHER condition is true: successful logins OR failed logins. Useful for getting a complete authentication picture.
NOT Exclude a condition
EventCode=4624 NOT user=Administrator
Returns successful logins, but excludes any event where the user is Administrator. Useful for filtering out expected/admin activity.

These three operators — AND, OR, NOT — combined with field filters form the foundation of every SOC investigation.

Knowledge Check

Test Your Understanding

These questions verify you can think like a SOC analyst when searching and investigating logs.

1. What does SPL stand for?

Correct! SPL stands for Search Processing Language. It is the query language used to search, filter, and analyze data in Splunk.

2. Why do SOC analysts search logs instead of reading them manually?

Correct! A single enterprise generates 500,000+ events daily. No human can read them all. Searching is the only way to find relevant events at scale.

3. What is the difference between Event ID 4624 and Event ID 4625?

Correct! Event ID 4624 = successful logon. Event ID 4625 = failed logon. These are the two most commonly used Event IDs in SOC investigations.

4. Can a single failed login (4625) be considered an attack?

Correct! A single failed login is often a mistyped password. But 50 failures in 2 minutes from a foreign IP is an attack. Context is everything.

5. What is a false positive?

Correct! A false positive is an alert that triggers on benign activity. Example: a user mistyping their password triggers a "failed login" alert, but it is not an attack.

6. What is the purpose of filtering in SPL searches?

Correct! Filtering removes irrelevant data, allowing analysts to focus on the specific events that matter for their investigation.

7. Why is the Time Picker important during investigations?

Correct! Setting the correct time range is essential. Searching all time for an event that happened at 10:15 AM adds unnecessary noise. Good analysts narrow the timeframe first.

8. What SPL query would you use to find all successful Administrator logins?

Correct! EventCode=4624 user=Administrator filters for successful logins where the user is Administrator — a precise and targeted search.

9. 50 failed logins in 2 minutes from a single IP. What does this pattern suggest?

Correct! 50 failures in 2 minutes from a single IP is a classic brute-force pattern. The attacker is rapidly trying different passwords against the same account.

10. What is the correct mindset when an alert fires?

Correct! Alert ≠ Incident. Every alert requires investigation. Gather context (who, what, when, where, why), then determine if the activity is malicious or benign.
